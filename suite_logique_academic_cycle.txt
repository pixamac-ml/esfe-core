SUITE LOGIQUE APRES LA FONDATION academic_cycle

La suite logique, c'est de passer de "fondation technique" a "workflow utilisable" sans tout brancher d'un coup.

Priorite recommandee :

1. Creer les cycles par annexe pour l'annee en cours

Dans l'admin, cree un BranchAcademicCycle pour chaque annexe + annee academique actuelle.
Exemple : Kalaban Coura / 2025-2026 / active.

2. Brancher un vrai bouton dashboard staff/DG

Actuellement les URLs existent, mais ce n'est pas encore integre proprement dans les dashboards existants.
Il faut ajouter une entree "Cycle academique" dans le dashboard DG / directeur des etudes / gestionnaire.

3. Fiabiliser le rapport readiness

Le service existe deja, mais il faut l'adapter plus finement a tes vraies regles :
- semestre finalise ou publie,
- toutes les notes saisies,
- bulletins generes,
- anomalies bloquantes,
- classe prete pour deliberation.

4. Implementer le workflow reel de cloture

Une annexe doit pouvoir :
- generer son rapport,
- corriger les blocages,
- demarrer la deliberation,
- cloturer uniquement si tout est pret,
- ecrire les logs.

5. Brancher les decisions etudiantes

Apres cloture :
- promu,
- promu avec dette academique,
- redoublant,
- diplome,
- abandon,
- transfert.

La, il faut relier le calcul aux vraies moyennes/resultats deja presents dans academics.

6. Preparer la reinscription

Une fois les decisions generees :
- creer les AcademicReEnrollment,
- generer les tokens,
- afficher la page pre-rentree cote etudiant,
- bloquer doucement l'acces si la reinscription n'est pas finalisee.

7. Brancher paiement minimum + activation dashboard

Quand un ancien etudiant paie le minimum :
- marquer la reinscription comme paiement valide,
- activer la reinscription,
- appliquer StudentAccessPolicy,
- basculer l'etudiant vers son nouveau contexte academique.

Prochaine etape de developpement recommandee :

Integrer le module dans les dashboards existants + rendre le rapport readiness vraiment exploitable visuellement.

Apres ca, on peut faire le workflow complet :
readiness -> deliberation -> cloture -> decisions -> reinscriptions.

- class_reports_overview_view,
- annual_class_report_view,
- student_annual_report_view,
- student_year_report_view.

Risque :
acces direct par URL a des releves publies d'autres etudiants/classes.

3. Import Excel de notes trop permissif

academics/views_import.py bloque seulement les informaticiens hors annexe, mais les autres utilisateurs connectes ne sont pas explicitement filtres par role.

Il faut limiter clairement a :
- DG / direction,
- directeur des etudes,
- informaticien autorise,
- eventuellement staff academique autorise.

Avec filtrage annexe obligatoire.

4. Ancien service surveillance incoherent avec la nouvelle architecture

academics/services/surveillance.py construit encore un dashboard base sur SurveillanceNote / ClassPresenceCheck.

Or le projet a maintenant :
- LessonLog,
- StudentAttendance,
- TeacherAttendance,
- StudentCase.

Il faut refaire ce service ou le retirer.

5. Workflow semestre disperse

La logique de changement de statut semestre existe a plusieurs endroits :
- academics.services.workflow,
- portal.views.views,
- portal.services.notes_workflow,
- portal.views.admin_grades.

Il faut centraliser dans un service unique pour eviter les incoherences :
DRAFT -> NORMAL_ENTRY -> NORMAL_LOCKED -> RETAKE_ENTRY -> FINALIZED -> PUBLISHED.

Verifications faites :

- python manage.py check : OK.
- python manage.py makemigrations --check --dry-run : OK.
- Import global services : seul academics.services.surveillance echoue.
- Tests complets non relances, car PostgreSQL n'a pas le droit de creer une base de test.

Suite logique recommandee :

Commencer par corriger les 3 points bloquants :

1. Nettoyer/remplacer academics.forms et academics.services.surveillance.
2. Securiser les vues de rapports.
3. Verrouiller l'import Excel par role + annexe.

Ensuite on pourra terminer proprement le workflow :
notes -> semestres -> rattrapage -> publication -> rapports.
