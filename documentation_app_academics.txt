1. ROLE GENERAL DE L'APP
------------------------

L'app "academics" porte le coeur pedagogique et academique du projet ESFE.
Elle ne gere pas seulement une liste de cours : elle fait le lien entre les
inscriptions administratives, les classes reelles, les semestres, les UE, les EC,
les notes, les credits, les bulletins, les diplomes, les emplois du temps et le
cahier de texte.

Elle sert donc a transformer une inscription administrative valide en parcours
academique exploitable :

- rattacher un etudiant a une classe academique concrete ;
- structurer une classe en semestres, UE et EC ;
- saisir ou importer les notes par EC ;
- calculer les credits, moyennes et decisions ;
- generer des releves, bulletins et documents de diplome ;
- planifier les cours et detecter les conflits d'emploi du temps ;
- suivre l'execution des seances et le cahier de texte ;
- alimenter les dashboards direction, surveillance, portail etudiant et autres
  apps connectees.

3. MODELE DE DONNEES
--------------------

3.1. Referentiels simples

Language
- Stocke les langues disponibles.
- Champs : name, code, is_active, created_at.

Profession
- Stocke les metiers/professions.
- Champs : name, description, is_active, created_at.


3.2. Annee academique

AcademicYear
- Reference centrale d'une annee academique, par exemple "2025-2026".
- Champs : name, start_date, end_date, is_active, created_at.
- Validation :
  - start_date doit etre avant end_date ;
  - une seule annee academique peut etre active a la fois.
- Le save() appelle full_clean(), donc ces controles s'appliquent avant
  enregistrement.


3.3. Classe academique

AcademicClass
- Represente une vraie classe pedagogique : programme + annexe + annee + niveau.
- Champs principaux :
  - programme : lien vers formations.Programme ;
  - branch : annexe ;
  - academic_year : annee academique ;
  - level : L1, L2, L3, M1, M2, etc. ;
  - study_level : DEF, BAC, LICENCE, MASTER ;
  - validation_threshold : seuil de validation, souvent 10 ou 12 ;
  - is_active, is_archived, archived_at.
- Contrainte :
  - une seule classe pour le meme programme, la meme annexe, la meme annee et le
    meme niveau.
- display_name retourne le nom lisible de la classe.


3.4. Inscription academique

AcademicEnrollment
- C'est le pont entre l'inscription administrative et le suivi academique.
- Elle relie :
  - une Inscription administrative ;
  - un utilisateur etudiant ;
  - un Programme ;
  - une Branch ;
  - une AcademicYear ;
  - une AcademicClass.
- Statuts :
  - ACTIVE ;
  - ARCHIVED ;
  - COMPLETED ;
  - SUSPENDED ;
  - TRANSFERRED ;
  - ABANDONED.
- Validation :
  - l'inscription doit avoir un paiement valide, etre partiellement payee/active
    ou deja posseder un etudiant cree ;
  - programme, annexe et classe doivent rester coherents avec la candidature ;
  - la classe cible doit etre active et non archivee ;
  - un etudiant ne peut pas avoir deux inscriptions academiques actives pour le
    meme programme et la meme annee.
- save() synchronise status avec is_active/is_archived/archived_at.


3.5. Semestre, UE et EC

Semester
- Rattache un semestre a une AcademicClass.
- Champs :
  - academic_class ;
  - number : 1 ou 2 ;
  - status ;
  - total_required_credits, par defaut 30.
- Statuts de workflow :
  - DRAFT ;
  - NORMAL_ENTRY ;
  - NORMAL_LOCKED ;
  - RETAKE_ENTRY ;
  - FINALIZED ;
  - PUBLISHED.
- Contrainte :
  - une classe ne peut avoir qu'un semestre 1 et un semestre 2.
- calculate_for_enrollment() appelle compute_semester_result().

UE
- Unite d'enseignement rattachee a un semestre.
- Champs : semester, code, title.
- Les credits et coefficients d'une UE sont calcules depuis ses EC.
- Contrainte : code unique dans un semestre.
- calculate_for_enrollment() appelle compute_ue_result().

EC
- Element constitutif, c'est-a-dire la matiere evaluee.
- Champs : ue, title, credit_required, coefficient.
- Validation :
  - credit_required et coefficient doivent etre positifs ;
  - credit_required ne doit pas depasser 6 ;
  - credit_required ne doit pas etre inferieur au coefficient.


3.6. Contenus pedagogiques

ECContent

3.7. Notes

4. CALCULS ACADEMIQUES
----------------------

4.1. Seuil de validation

Le service academics/services/grading.py contient resolve_threshold().

Priorite :
1. utiliser validation_threshold de la classe ;
2. sinon utiliser study_level :
   - DEF -> 10 ;
   - BAC -> 12 ;
   - LICENCE -> 12 ;
   - MASTER -> 12 ;
3. fallback global -> 10.


4.2. Calcul d'un EC

calculate_ec_grade(note, coefficient, credit_required, threshold) :

- convertit proprement les valeurs en Decimal ;
- borne la note entre 0 et 20 pour le calcul ;
- calcule note_coefficient = note * coefficient ;
- donne tous les credits de l'EC si note >= threshold ;
- sinon donne 0 credit ;
- retourne note, note_coefficient, credit_obtained, is_validated.

apply_ec_grade(instance) :

- choisit final_score :
  - retake_score si seul le rattrapage existe ;
  - normal_score si seul le normal existe ;
  - max(normal_score, retake_score) si les deux existent ;
- applique calculate_ec_grade ;
- met a jour note, note_coefficient, credit_obtained et is_validated.


4.3. Calcul d'une UE

compute_ue_result(ue, enrollment) :

- recupere tous les EC de l'UE ;
- recupere les notes ECGrade de l'etudiant ;
- calcule la moyenne UE :
  somme(note_coefficient EC) / somme(coefficients EC) ;
- calcule les credits obtenus ;
- detecte les notes manquantes ;
- liste les EC echoues ;
- retourne un statut :
  - incomplete ;
  - validated ;
  - failed.

Une UE est validee si :

- toutes les notes attendues sont presentes ;
- la moyenne UE atteint le seuil ;
- les credits obtenus couvrent les credits requis de l'UE.


4.4. Calcul d'un semestre

compute_semester_result(semester, enrollment) :

- calcule toutes les UE du semestre ;
- calcule la moyenne semestrielle :
  somme(moyenne UE * coefficient UE) / somme(coefficients UE) ;
- cumule credits requis et credits obtenus ;
- detecte les notes manquantes ;
- liste les UE/EC echoues ;
- calcule un pourcentage de credits obtenus ;
- produit des raisons de blocage.

Un semestre est valide si :

- il est complet ;
- la moyenne atteint le seuil ;
- tous les credits requis sont obtenus.

compute_class_ranking() classe les etudiants d'une classe pour un semestre avec
gestion simple des ex-aequo.


4.5. Calcul annuel et decision

compute_annual_result(enrollment) :

- calcule tous les semestres de la classe ;
- pondere la moyenne annuelle par les credits requis ;
- cumule credits requis/obtenus ;
- remonte les blocages ;
- retourne complete/validated/failed.

compute_annual_decision(enrollment) :

- determine la decision academique automatique :
  - promoted ;
  - promoted_with_debt ;
  - repeated ;
  - completed.
- Regles principales :
  - tous les semestres valides -> passage ou completion si niveau terminal ;
  - aucune validation -> redoublement ;
  - un seul semestre echoue peut etre compense si l'ecart sous le seuil ne
    depasse pas DEFAULT_SEMESTER_COMPENSATION_MAX_GAP = 0.50 ;
  - une dette academique n'est pas autorisee en niveau terminal L3 ou M2 ;
  - notes incompletes -> decision bloquee.

compute_year_result(student, academic_year) :

- resout l'etudiant et l'annee ;
- retrouve l'inscription academique active ;
- calcule S1, S2, resultat annuel et decision ;
- sert aux rapports annuels et vues etudiant.


5. AFFECTATION ACADEMIQUE D'UN ETUDIANT
---------------------------------------

Le service academics/services/enrollment_service.py expose
assign_student_academic_enrollment(inscription).

Objectif :
- creer automatiquement AcademicEnrollment apres une inscription administrative.

Priorite :
1. si inscription.academic_class est deja renseignee, elle est validee via
   validate_academic_class_for_candidature() ;
2. sinon le systeme resout automatiquement le contexte via la candidature :
   programme + annexe + annee academique + entry_year -> level -> classe.

Le resolveur academics/services/academic_context_resolver.py fait ce travail :

- resolve_academic_year() transforme une valeur legacy/string en AcademicYear ;
- resolve_level_from_entry_year() transforme entry_year en L1/L2/L3/M1/M2 selon
  le cycle ;
- resolve_academic_context() cherche ensuite l'AcademicClass active compatible.

Si une donnee manque ou si plusieurs classes correspondent, le service retourne
un statut de type manual_required_* au lieu de creer une inscription academique
incorrecte.

Le service academics/services/academic_positioning.py aide les ecrans de
positionnement :

- liste les classes disponibles pour une candidature ;
- trie les niveaux ;
- calcule les frais par niveau ;
- valide qu'une classe choisie correspond bien a la candidature.

- endpoint POST : imports/grades/upload/ ;

- lit le fichier avec pandas ;
- mappe les colonnes aux EC ;
- retrouve l'etudiant par ENROLLMENT_ID, MATRICULE ou NOM/PRENOM ;
- ignore les cellules vides ;
- refuse les notes hors 0-20 ;
- si des etudiants sont inconnus ou des notes invalides existent, il retourne
  les problemes sans appliquer les updates ;
- sinon il fait update_or_create sur ECGrade puis apply_ec_grade().

Resultat retourne :
- updated ;
- skipped_empty ;
- skipped_unknown_columns ;
- skipped_unknown_students ;
- skipped_invalid_scores ;
- unknown_columns ;
- student_issues.

- Candidature fournit programme, annexe, annee academique et entry_year.

students
- Student fournit le profil etudiant, matricule et donnees personnelles.
- TeacherAttendance est utilise pour detecter les enseignants absents.

accounts

  academics.

payments

14. FLUX METIER TYPIQUES
------------------------

14.1. Creation d'un parcours etudiant

1. Une candidature existe dans admissions.
2. Une inscription administrative est creee dans inscriptions.
3. Apres paiement ou activation, assign_student_academic_enrollment() est appele.
4. Le service trouve la classe academique compatible.
5. AcademicEnrollment est cree.
6. L'etudiant peut apparaitre dans les rapports, imports de notes, planning et
   portail etudiant.


14.2. Saisie/import des notes

1. Le responsable telecharge un template pour classe + semestre.
2. Il remplit les notes.
3. Il upload le fichier.
4. Le service mappe etudiants et EC.
5. Les ECGrade sont crees ou mis a jour.
6. apply_ec_grade calcule note finale, credits et validation.
7. Les rapports utilisent compute_semester_result() et compute_annual_result().


14.3. Publication des resultats

1. Les notes sont completes.
2. Le semestre avance dans son workflow jusqu'a PUBLISHED.
3. Les releves semestriels deviennent disponibles.
4. Les bulletins semestriels peuvent etre generes.
5. Quand tous les semestres sont publies, le bulletin annuel devient disponible.
6. Si le cycle est termine et valide, le diplome peut etre prepare.


14.4. Planification d'une semaine

1. Des creneaux hebdomadaires sont crees via WeeklyScheduleSlot.
2. Le surveillant/direction materialise les evenements dates de la semaine.
3. create_schedule_event() verifie conflits, coherence et volumes horaires.
4. Les dashboards affichent la grille semaine.
5. start_schedule_event() et complete_schedule_event() suivent l'execution.
6. Les logs de changement conservent l'historique.


14.5. Cahier de texte

1. Un cours est planifie dans AcademicScheduleEvent.
2. Apres ou pendant le cours, un LessonLog est cree.
3. Si TeacherAttendance indique une absence, le log passe en absent_teacher.
4. Le dashboard peut detecter les cours planifies sans cahier de texte.
5. Les controles quotidiens remontent les anomalies.


15. POINTS D'ATTENTION TECHNIQUES
---------------------------------

- Plusieurs validations sont dans les modeles avec full_clean() appele au save().
  Il faut donc respecter les coherences metier avant d'enregistrer.

- Les calculs sont en Decimal, ce qui evite les erreurs de float pour les notes
  et credits.

- Les bulletins stockent un snapshot JSON : cela fige les donnees au moment de la
  generation, meme si les notes changent plus tard.

- Les rapports PDF dependent de WeasyPrint. Si WeasyPrint ou ses dependances
  systeme manquent, la vue retourne une erreur 500 explicite.

- Les permissions sont fortement liees aux positions accounts et a l'annexe de
  l'utilisateur.

- Le service d'emploi du temps depend aussi de portal.models.DirectorTeacherAssignment.
  Une planification peut donc echouer meme si les champs de l'event sont valides,
  si l'enseignant n'est pas affecte ou depasse son volume horaire.

- Le modele AcademicEnrollment reste le pivot. Sans lui, un etudiant ne peut pas
  etre correctement calcule, classe, exporte ou rattache aux rapports.


16. RESUME COURT
----------------

L'app "academics" est le moteur pedagogique du systeme ESFE. Elle part des
inscriptions administratives et construit tout le suivi academique : classes,
semestres, UE/EC, notes, credits, decisions, bulletins, diplomes, emplois du
temps, execution des cours et cahier de texte. Sa logique est principalement
dans academics/services/, tandis que models.py definit les objets persistants et
views.py expose les rapports, PDF et APIs JSON.
