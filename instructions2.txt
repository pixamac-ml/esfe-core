# SPÉCIFICATIONS TECHNIQUES : INTERFACE INSTANTANÉE (DASHBOARD & TOASTS)

## 1. OBJECTIF
Transformer notre système de gestion universitaire pour obtenir une interface "Instantanée" (Optimistic UI). L'utilisateur ne doit jamais percevoir de latence, de blocage ou d'attente lors des actions courantes (ex: validation d'une note, inscription d'un étudiant, modification d'un statut).

---

## 2. DÉFINITION ET RÔLES

### A. Le Toast (Notification Éphémère)
* **Nature** : Composant UI purement visuel géré côté client par Alpine.js et Tailwind CSS.
* **Comportement** : Il apparaît immédiatement au clic de l'utilisateur (temps de réponse < 50ms) pour confirmer visuellement que l'action est prise en compte, puis disparaît après 3 secondes.
* **Rôle** : Il supprime la perception d'attente (illusion de vitesse).

### B. Le Dashboard Instantané (Mise à jour Optimiste)
* **Nature** : Gestion d'état asynchrone synchrone-visuel.
* **Comportement** : Dès qu'une action est déclenchée, l'interface du Dashboard (compteurs, listes, tableaux) se met à jour instantanément à l'écran via Alpine.js, SANS attendre la réponse HTTP du serveur Django.
* **Rôle** : Fluidité absolue de la navigation.

---

## 3. ARCHITECTURE TECHNIQUE DU FLUX INSTANTANÉ

Toute action utilisateur (Ex: Bouton "Inscrire l'étudiant") doit suivre ce flux parallèle et simultané :

## 4. INSTRUCTIONS DE CODAGE POUR L'IA

### Étape 1 : Le Toast Centralisé (À intégrer dans base.html)
Injecte ce composant global pour écouter les notifications de l'application :
```html
<div x-data="{ open: false, message: '', type: 'success' }"
     x-on:notify.window="message = event.detail.message; type = event.detail.type || 'success'; open = true; setTimeout(() => open = false, 3000)"
     x-show="open"
     x-transition
     class="fixed bottom-5 right-5 px-4 py-3 rounded-lg shadow-lg z-50 flex items-center text-white font-medium"
     :class="type === 'success' ? 'bg-emerald-600' : 'bg-rose-600'"
     style="display: none;">
    <span x-text="message"></span>
</div>
```

### Étape 2 : Bouton d'action optimiste dans un Django Component
Applique cette logique à nos boutons de gestion universitaire (ex: valider une inscription) pour mettre à jour l'interface avant la réponse du serveur :
```html
<div x-data="{ inscrit: false, totalEtudiants: 10 }">
    <!-- Le compteur se met à jour immédiatement via Alpine.js -->
    <div class="text-xl font-bold">
        Étudiants inscrits : <span x-text="inscrit ? totalEtudiants + 1 : totalEtudiants">10</span>
    </div>

    <!-- Le bouton déclenche le Toast ET la mise à jour UI avant que HTMX ne termine -->
    <button hx-post="{% url 'inscrire_etudiant' student_id=student.id %}"
            hx-swap="none"
            x-on:click="
                inscrit = true;
                \$dispatch('notify', { message: 'Étudiant inscrit instantanément !', type: 'success' })
            "
            class="mt-2 bg-blue-600 hover:bg-blue-700 text-white font-semibold py-2 px-4 rounded transition-colors">
        Inscrire l'étudiant
    </button>
</div>
```

### Étape 3 : Côté Django (Python)
Les vues appelées par HTMX doivent être ultra-légères. Si le traitement prend du temps (ex: génération de bulletin de notes PDF, envoi d'email de confirmation), sépare la logique :
1. Sauvegarde rapide en base de données ou lancement de tâche en tâche de fond (*Background Task* via Threads ou Celery).
2. Retour immédiat d'une réponse `HttpResponse(status=204)` ou `200 OK` avec un corps vide pour libérer HTMX sans surcharger le réseau.

## 5. RÉSULTAT ATTENDU POUR LA PRÉSENTATION
Pendant la démo devant le jury, lorsque je clique sur n'importe quelle action du système universitaire, le Dashboard doit réagir au quart de tour (les chiffres changent, le toast s'affiche). La communication avec Django se fait de manière transparente et invisible en arrière-plan.

Applique cette stratégie d'interface optimiste sur l'ensemble des formulaires et boutons de notre application.
