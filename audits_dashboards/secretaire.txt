================================================================================
AUDIT CRITIQUE - DASHBOARD SECRETAIRE (app `secretary/`)
================================================================================
Perimetre : app Django `secretary/` (models.py, views.py, selectors.py,
services.py, forms.py, urls.py, permissions.py, signals.py, admin.py,
tests.py) + templates `secretary/templates/secretary/` (dashboard.html,
base.html, listes, formulaires, partials HTMX).

Clarification prealable importante :
- Le role "secretaire" (position="secretary", groupe "secretary"/"secretaries")
  est mappe vers le role canonique `staff_admin` (accounts/access.py).
- A la connexion, `accounts/views.py` redirige la secretaire vers
  `accounts_portal:portal_secretary`.
- Mais `portal/views/views.py` -> `secretary_portal()` n'est qu'un STUB :
  `return redirect("accounts_portal:portal_dashboard")`. Il n'existe AUCUN
  dashboard secretaire reel sous `accounts/dashboards/` ou
  `accounts/templates/accounts/dashboard/`.
- Le VRAI dashboard secretaire, complet et fonctionnel, est l'app autonome
  `secretary/` montee sur `/secretary/` (vue `secretary_dashboard`,
  template `secretary/templates/secretary/dashboard.html`).
=> Cet audit porte sur `/secretary/`, qui est la dashboard reellement
   utilisable par une secretaire. Le stub `/portal/secretary/` est un
   probleme de routage en soi (voir section 2).


================================================================================
1. CE QUI MARCHE VRAIMENT
================================================================================

[1.1] Filtrage par annexe systematique
--------------------------------------
`secretary/selectors.py` centralise la portee via `_apply_branch_scope()`
(lignes ~30-75) et est appele par TOUTES les requetes principales :
- `get_registry_queryset` (RegistryEntry a un champ `branch` direct)
- `get_appointments_queryset`, `get_visits_queryset`/`get_active_visits_queryset`
  /`get_today_visits_queryset`, `get_documents_queryset`, `get_tasks_queryset`
  (ces 4 modeles n'ont pas de FK `branch` direct : la portee passe par
  `created_by__profile__branch` / `received_by__profile__branch`)
- `get_active_students` / `search_students` / `get_classes_queryset` /
  `get_today_appointments_queryset` filtrent via
  `inscription__candidature__branch=resolved_branch`.

Couverture testee explicitement dans `secretary/tests.py`, classe
`ScopeTests` (lignes 646-727) : 5 tests verifient qu'une entree creee par
`other_secretary` (autre annexe) est invisible pour `self.secretary`, pour
RegistryEntry, Appointment, VisitorLog, DocumentReceipt, SecretaryTask.
=> C'est un point FORT, nettement mieux couvert par les tests que dans
   d'autres dashboards audites.

[1.2] CRUD reel avec validation metier (`secretary/services.py`)
------------------------------------------------------------------
Toutes les operations cle sont des fonctions de service `@transaction.atomic`
qui appellent `full_clean()` via un helper `_clean_instance` :
- Registre : `create_registry_entry`, `update_registry_entry`,
  `start_registry_entry_processing`, `mark_registry_processed`,
  `archive_registry_entry` -> generation automatique de
  `registry_number` (format `ESFE-<CODE_ANNEXE>-...`), `daily_number`,
  `event_identifier`, et historique JSON (`_history_event`).
- Rendez-vous : `create_appointment`, `update_appointment`,
  `complete_appointment` -> avec `prevent_conflicts()` qui empeche deux
  RDV qui se chevauchent pour la meme personne/le meme crenaau.
- Visiteurs : `register_visitor`, `update_visitor`, `close_visit`
  (renseigne `departed_at` automatiquement).
- Documents : `register_document`, `update_document`, `archive_document`,
  `document_receipt_start` (passe en "en cours").
- Taches : `create_task`, `update_task`, `complete_task`,
  `task_start`/`assign_task`.

Toutes ces transitions sont gardees par des regles de coherence (ex :
`archive_registry_entry` leve `ValidationError` si deja archive,
`complete_appointment` leve si deja termine, etc.) - testees une a une
dans `tests.py` (RegistryEntryServiceTests, AppointmentServiceTests,
VisitorServiceTests, DocumentServiceTests, TaskServiceTests).

[1.3] Routage intelligent du registre (REGISTRY_ROUTING_RULES)
------------------------------------------------------------------
`secretary/services.py` definit `REGISTRY_ROUTING_RULES` : un dictionnaire
qui mappe chaque `entry_type` (visite parent, paiement scolarite, depot
colis, retrait diplome, reclamation, etc.) vers un `target_service`, une
`priority`, un `workflow_code` et une liste d'`actions`.
`apply_registry_routing()` applique ces regles automatiquement a la
creation/mise a jour d'une entree de registre.
Le modal de creation (`secretary/templates/secretary/modals/form_modal.html`,
lignes 83-115) injecte ces regles via `json_script` et un petit script JS
qui met a jour en direct les champs `target_service`/`priority` ET un
panneau d'apercu ("Service cible", "Workflow", "Actions prevues") quand la
secretaire change le type d'entree. C'est une vraie logique metier
dynamique, pas de la decoration.

[1.4] Export PDF/XLSX reel du registre quotidien
------------------------------------------------------------------
`secretary/views.py`, fonction `daily_registry_report` (lignes 1139-1215) :
- `?format=xlsx` -> genere un classeur openpyxl reel, en-tete style,
  colonnes (numero, type, visiteur, motif, statut, service cible, priorite,
  heure...), filtre par annexe et par date.
- `?format=pdf` -> genere un PDF reportlab (SimpleDocTemplate, format A4
  paysage, TableStyle avec entetes colorees).
- Sans parametre -> page HTML `daily_registry_report.html` avec les memes
  donnees + boutons d'export + bouton impression navigateur.
=> Contrairement a d'autres dashboards de ce projet ou les boutons
   "Exporter" ouvrent juste un panneau HTML, ici l'export produit de
   vrais fichiers telechargeables.

[1.5] Pourcentages "Repartition par type" reellement calcules
------------------------------------------------------------------
`secretary/views.py`, lignes 246-257 :
```python
registry_qs = get_registry_queryset({"archived": False, "active_only": True}, user=request.user, branch=branch)
type_counts = registry_qs.values("entry_type").annotate(count=Count("id")).order_by("-count")
total = sum(tc["count"] for tc in type_counts) or 1
context["registry_type_distribution"] = [
    {... "pct": round(tc["count"] / total * 100) ...}
    for tc in type_counts
]
```
Le `total = ... or 1` evite la division par zero. Ces pourcentages sont
affiches dans le panneau "Repartition par type" du dashboard et sont
intregalement issus de la base, par annexe. Bon contre-exemple aux
pourcentages fixes trouves dans d'autres audits (ex. DG : 39%/20%/15%
de repartition de depenses codes en dur).

[1.6] Boutons d'action reels presents dans plusieurs listes
------------------------------------------------------------------
- `appointment_list.html` (lignes 49-57) : "Modifier" (ouvre formulaire
  HTMX en modal) + "Cloturer" (POST reel vers `appointment_complete`).
- `document_receipt_list.html` (lignes 48-61) : "Modifier", "Prendre en
  charge" (POST `document_receipt_start`), lien "Fichier" si piece jointe,
  "Archiver" (POST `document_receipt_archive`).
- `dashboard_overview_registry.html` : bouton "Traiter" reel (POST
  `registry_mark_processed`) avec retrait optimiste de la ligne
  (`data-sg-optimistic="fade-remove"`).
- `dashboard_visits_open.html` (lignes 18-20) : bouton "Sortie" reel (POST
  `visitor_complete`), pagination fonctionnelle.
- `class_students.html` / `student_results.html` : boutons "Dossier"
  (ouvre le tiroir `student_drawer.html` via `student_snapshot`) et
  "Registre" (pre-remplit une creation de RegistryEntry liee a l'etudiant)
  - tous deux reels et cables.

[1.7] Fiche etudiant ("Dossier") riche et reelle
------------------------------------------------------------------
`secretary/services.py`, `get_student_snapshot()` (lignes 435-462) +
`_get_secretary_student_schedule()` (465-513) +
`_student_presence_label`/`_student_presence_tone` (516-539) construisent
un snapshot complet :
- identite, matricule, classe, annee academique, annexe (via la chaine
  `inscription -> candidature`)
- compteurs d'activite administrative (registre, RDV, visites, pieces)
- emploi du temps hebdomadaire reel via
  `academics.services.schedule_service.get_student_week_schedule`
- statut de presence courant via `students.models.StudentAttendance`
  ("en cours", "absent", "en retard"...).
Le template `student_drawer.html` affiche tout cela + 3 boutons d'action
rapides (Nouvelle entree registre, Rendez-vous, Visite) tous fonctionnels
et pre-remplis avec l'ID etudiant.

[1.8] Couverture de tests solide
------------------------------------------------------------------
`secretary/tests.py` (727 lignes) couvre :
- Modeles (RegistryEntry, Appointment, VisitorLog, DocumentReceipt,
  SecretaryTask) : creation minimale, validations (`full_clean`), `__str__`,
  auto-titre, archivage.
- Services : creation/mise a jour/transition d'etat pour les 5 domaines,
  y compris les cas d'erreur (double archivage, double cloture, RDV dans
  le passe).
- Vues d'edition (Registry/Appointment/Visitor/Document/Task) : GET 200,
  POST met a jour la DB, redirection 302 vers la liste, 403 pour
  utilisateur non secretaire, 404 pour secretaire d'une autre annexe
  (RegistryEditViewTests.test_other_branch_secretary_cannot_access).
- Permissions (`is_secretary`, `ensure_secretary_access`) : superuser,
  groupe, profil, utilisateur normal.
- Portee/filtrage par annexe (5 tests, voir 1.1).
C'est la couverture de tests la plus complete observee parmi les
dashboards audites jusqu'ici (DG, enseignant).

[1.9] Formulaires reels (pas de simulacres)
------------------------------------------------------------------
`secretary/forms.py` : `RegistryEntryForm`, `AppointmentForm`,
`VisitorLogForm`, `DocumentReceiptForm`, `SecretaryTaskForm` sont des
ModelForm reels, avec querysets de FK scopes par annexe (ex. `assigned_to`

timezone, et `full_clean()`. `document_receipt_form.html` gere correctement
l'upload de fichier (`enctype="multipart/form-data"`, `hx-encoding`).

PROBLEME #10 - Resolution de portee fragile pour 4 modeles sur 5
------------------------------------------------------------------------------
Seul `RegistryEntry` a un champ `branch` FK direct (confirme dans
`secretary/migrations/0002_registryentry_attachment_registryentry_branch_
and_more.py`). Les 4 autres modeles (Appointment, VisitorLog,
DocumentReceipt, SecretaryTask) sont scopes via
`created_by__profile__branch` ou `received_by__profile__branch`
(`_apply_branch_scope` dans selectors.py).

=> Consequence pratique : si un enregistrement est cree par un utilisateur
   dont le `profile.branch` est NULL (compte mal configure, import de
   donnees historiques, compte superviseur sans annexe...), cet
   enregistrement devient invisible pour TOUTE secretaire scopee par
   annexe - y compris celle qui devrait normalement le traiter. Il n'y a
   pas de filet de securite (ex: fallback sur un champ `branch` propre a
   l'enregistrement). Le risque est mineur en usage normal (la secretaire
   est generalement `created_by`/`received_by` elle-meme et a un profil
   correctement configure), mais devient un point de friction silencieux
   en cas de comptes mal configures ou de creation par un autre role
   (ex. `assign_task` peut assigner une tache creee par un gestionnaire
   d'une autre annexe ?).

6. VERDICT
================================================================================

Le dashboard secretaire (`/secretary/`) est, parmi les dashboards audites
dans ce projet, celui dont l'architecture est la PLUS SOLIDE cote backend :
couche service propre avec `@transaction.atomic` + `full_clean()`, regles
metier reelles (anti-conflit de RDV, routage intelligent du registre,
numerotation automatique), filtrage par annexe systematique et TESTE, et
un export PDF/XLSX qui produit reellement des fichiers exploitables (pas
une simulation).

Cependant, il n'est PAS pret pour une mise en production sans corrections,
pour des raisons d'UX et de routage plutot que de fond technique :

1. Le chemin de connexion standard pour une secretaire
   (`/portal/secretary/`) ne menerait PAS vers ce dashboard mais vers le
   portail generique - a verifier/corriger en priorite absolue, sinon le
   dashboard audite ici n'est simplement jamais atteint par l'utilisatrice
   cible.
2. Une donnee visuelle codee en dur (barres "Processus de remise") donnera
   une fausse impression de suivi en temps reel.
3. Plusieurs actions du quotidien (cloturer un RDV du jour, gerer une tache
   du jour, traiter un document en attente) ne sont PAS accessibles depuis
   les apercus du dashboard, obligeant a naviguer vers d'autres pages -
   ce qui va a l'encontre de l'objectif d'un "tableau de bord".
4. Aucune notification n'est generee automatiquement par les actions du
   secretariat (RDV assigne, tache assignee, document route) - tout le
   suivi inter-services reste manuel/hors-systeme.
5. Absence de toute visibilite sur les candidatures/inscriptions, alors que
   c'est une attente raisonnable du role.

En synthese : la fondation (modeles, services, tests, filtrage par annexe,
exports) est de tres bonne qualite et largement reutilisable. Le travail
restant est concentre sur le ROUTAGE D'ACCES, la SUPPRESSION DES DONNEES
STATIQUES, et le COMPLETEMENT DES APERCUS DU DASHBOARD avec les actions deja
disponibles ailleurs dans l'app - un effort cible et raisonnable plutot
qu'une refonte.

# Brief de refonte front-end — Dashboard « Poste secrétaire » (ESFé Core)

> **Destinataire :** Claude Code.
> **Rôle attendu :** designer/intégrateur front-end senior. Tu interviens sur un module **déjà fonctionnel** ; tu ne réécris pas le back-end. Tu élèves le front-end : système de design, ergonomie de saisie, workflow, micro-interactions.
> **Stack imposée :** Django (templates) + HTMX + Tailwind CSS + JavaScript vanilla léger (Alpine.js autorisé si déjà présent, sinon JS natif). **Pas de React, pas de SPA.**
> **Langue de l'interface :** français intégral. Aucun libellé technique anglais visible.
> **Contraintes non négociables :** code propre niveau production, aucune solution temporaire ni hack, sécurité et performance prioritaires, accessibilité de base garantie.

---

## 0. Lis ceci avant de toucher au code

Trois règles d'architecture conditionnent toute la refonte. Les violer casse la cohérence ou la sécurité.

1. **La logique de routage (`type d'entrée → service cible / priorité / workflow / actions`) est autoritative côté serveur.** Elle vit dans une config Python (dict) ou une méthode de modèle, exposée au front pour l'affichage instantané. Le front ne décide rien ; il reflète. À l'enregistrement, le serveur re-dérive et valide. Ne jamais faire confiance aux valeurs envoyées par le client pour le service ou la priorité dérivés.
2. **Les transitions d'état sont validées côté serveur.** Le modèle `SecretaryStatusMixin` n'a pas de machine à états. Définir une table de transitions autorisées et la faire respecter dans la vue (ou `clean()`/`save()`). Les boutons grisés côté front sont une commodité, pas une protection.
3. **Le champ `history` (JSON) doit être alimenté à chaque changement d'état ou action.** Sinon la timeline d'activité est vide et la traçabilité (argument de vente) ne tient pas. Utiliser un schéma d'événement fixe (voir §7).

---

## 1. Nature du produit (à garder en tête en permanence)

Ce n'est pas un tableau de bord de consultation. C'est un **poste d'accueil front-office à fort débit**, interrompu en continu (un parent arrive pendant qu'on saisit une réclamation). La secrétaire doit :

- saisir une entrée en **moins de 10 secondes**, sans quitter l'écran ;
- router automatiquement vers le bon service ;
- reprendre une saisie interrompue sans perdre le contexte ;
- continuer à travailler **même réseau/courant coupés** (mode dégradé déjà prévu — à préserver).

Conséquences directes pour le design : **vitesse > esthétique décorative**, **clavier de première classe**, **contexte toujours visible**, **densité maîtrisée**, **états et priorités lisibles d'un coup d'œil**.

---

## 2. Diagnostic de l'existant (ce qu'on garde, ce qu'on corrige)

**À conserver (points forts) :**
- Concept de **routage intelligent** par type d'entrée.
- **Mode dégradé** (impression canevas + ressaisie). Penser à garder cette fonction visible et robuste.
- **Processus de remise** des pièces en 3 étapes.
- Fiche étudiant **en drawer latéral large** (déjà le bon pattern — à généraliser).
- Pastilles de contexte en haut (`0 entrées à router`, `0 visites ouvertes`) : bonne idée, à rendre vivantes.

**À corriger (par ordre de priorité) :**
1. **Libellés de formulaire en anglais** (`Title`, `Entry type`, `Visitor name`, `Related staff`, `Student class label`, `Submitted by name`, `Target service`, `Status`, `Related registry`, `File`…). Bloquant pour une démo commerciale. → §8.
2. **Routage intelligent statique** dans le formulaire (Service cible / Workflow / Actions prévues restent vides). → §6.
3. **Navigation dupliquée** : sidebar verticale + barre d'onglets horizontale listent les mêmes sections. → §9.
4. **Pas de système d'états/priorités** cohérent ; cartes KPI au style variable d'une page à l'autre ; tableaux sans badge de statut, sans pastille de priorité, sans action de ligne claire ; priorité « immédiate » invisible. → §4, §5.
5. **Esthétique non signée** (Tailwind par défaut). → §3.
6. **Formulaires longs sans divulgation progressive** : 12 champs affichés d'un bloc alors que le type d'entrée pourrait n'en exiger que 4. → §6.
7. **États vides plats** (« Aucune entrée en attente. ») sans appel à l'action. → §5.5.

---

## 3. Système de design (identité visuelle)

**Direction :** « le registre vivant ». L'interface est un registre administratif moderne. La signature mémorable n'est pas un dégradé ni une grande statistique : c'est le **traitement typographique des numéros de registre et des heures en monospace**, associé à un **flux chronologique du jour** qui sert de colonne vertébrale. On garde l'identité indigo existante, on la systématise.

> Éviter les trois clichés du design auto-généré : fond crème + serif contrasté + terracotta ; fond quasi noir + accent acid-green ; mise en page broadsheet à filets fins. Notre direction est institutionnelle, dense et lisible, pas « tendance ».

- **Heroicons uniquement** (tu l'as déjà en favori). Outline 20/24 pour la navigation et l'inline, Solid pour l'état actif et les pastilles de statut.
- **Supprimer tous les emojis** présents dans les titres (🏫 🎓 💵 📁…). Un seul système d'icônes, cohérent, monochrome teinté par le contexte.

## 5. Cœur métier — le formulaire contextuel « Routage intelligent »

C'est la pièce maîtresse. Le formulaire « Nouvelle entrée de registre » doit devenir **réactif et progressif**.

### 5.1 Comportement attendu

1. À l'ouverture du drawer, on demande d'abord **le type d'entrée**, présenté comme une **grille de cartes-type** (icône Heroicon + libellé FR), pas un `<select>` discret. C'est le premier geste, le plus visible.
2. À la sélection d'un type, **instantanément** :
   - la carte « Routage intelligent » s'allume : **Service destinataire**, **Priorité** (pré-réglée), **Workflow**, **Actions prévues** (pastilles) ;
   - le corps du formulaire **se reconfigure** : seuls les champs pertinents apparaissent, les requis s'ajustent, les placeholders s'adaptent (divulgation progressive) ;
   - les champs dérivés (service, priorité) sont **affichés en lecture**, modifiables seulement via une action explicite « Ajuster » (sinon on casse la promesse d'automatisation).
3. À l'enregistrement : succès → toast + nouvelle ligne **en tête du flux du jour** (UI optimiste, voir §10), drawer fermé.

### 5.2 Implémentation (instantanéité + sécurité)

Approche **hybride** :
- **Source de vérité serveur** : un dict `REGISTRY_ROUTING` (ou méthode `RegistryEntry.routing_for(entry_type)`) renvoyant `{service, priority, workflow_code, linked_actions, visible_fields, required_fields}`.
- **Instantanéité front** : exposer ce mapping une seule fois dans le gabarit via
  `<script type="application/json" id="registry-routing">{{ routing_map_json|safe }}</script>`
  consommé par un petit contrôleur JS/Alpine qui met à jour la carte et l'affichage des champs **sans aller-retour réseau** (crucial en réseau faible).
- **Variante full-HTMX** (si tu préfères ne pas dupliquer la logique en JS) : `hx-get="{% url 'secretary:registry_routing_preview' %}" hx-trigger="change" hx-target="#routing-card"` qui renvoie le partial carte + le partial des champs reconfigurés. Choisir **une** approche, pas les deux.
- **À l'enregistrement**, le serveur **re-dérive** service/priorité depuis le type (ignore toute valeur cliente non autorisée) et valide.

## 6. Workflow & transitions d'état

### 6.1 Cycle de vie (rappel modèle)
`pending` (En attente) → `in_progress` (En cours) → `transferred` (Transféré) → `completed` (Traité) ; sorties `cancelled` (Annulé) et `archived` (Archivé).

### 6.2 Transitions autorisées (à définir côté serveur ET refléter côté front)

| Depuis | Vers autorisé |
|---|---|
| En attente | En cours, Transféré, Annulé |
| En cours | Transféré, Traité, Annulé |
| Transféré | En cours, Traité, Annulé |
| Traité | Archivé |
| Annulé | Archivé |
| Archivé | — (terminal) |

Le front n'affiche que les boutons d'action correspondant aux transitions valides depuis l'état courant. Le serveur **refuse** toute transition hors table (HTTP 422 + message FR).

### 6.3 Vue Kanban (section Registre)
Proposer, en plus de la liste, une **vue Kanban** (colonnes = états : En attente · En cours · Transféré · Traité). Déplacement par bouton (et glisser-déposer optionnel). Chaque déplacement : POST HTMX → validation transition → écriture `history` → re-render de la carte. Filtre par priorité et service en tête.

### 6.4 Stepper dans le drawer détail
Dans le drawer d'une entrée : un **stepper horizontal** montrant l'état courant dans le cycle + boutons d'action contextuels + la **timeline `history`** dessous.

## 7. Traçabilité — schéma `history` et timeline

À chaque action, **append** un événement structuré dans `history` :

```json
{
  "at": "2026-06-13T13:23:00Z",
  "actor": "secretaire_mrb",
  "action": "status_change",
  "from": "pending",
  "to": "transferred",
  "service": "Gestionnaire",
  "note": "Routé vers gestionnaire pour paiement scolarité"
}
```

`timeline.html` rend ces événements en frise verticale (heure en mono, acteur, libellé d'action FR, transition d'état colorée). C'est la preuve de traçabilité que tu mets en avant commercialement — elle doit être systématiquement alimentée (cf. §0.3).

## 9. Navigation — supprimer la redondance

Aujourd'hui la sidebar verticale et la barre d'onglets horizontale listent les **mêmes** sections. Décision :

- **Garder la sidebar** comme navigation principale (groupes Accueil / Organisation / Communication déjà bons).
- **Supprimer la barre d'onglets horizontale** qui la duplique. Récupérer cet espace pour le flux du jour.
- La **barre de contexte** en haut (annexe, date, `entrées à router`, `visites ouvertes`) reste, mais devient **vivante** (compteurs live HTMX) et chaque pastille devient un **filtre cliquable**.
- Le titre de page + les actions de page (ex. « Nouvelle entrée », « Liste complète ») vivent dans un bandeau d'en-tête de section, pas dans une barre d'onglets.

Alternative si tu tiens à une navigation horizontale (ex. petits écrans) : la rendre **collapsible** et mutuellement exclusive avec la sidebar, jamais les deux affichées en plein en même temps.

## 10. Interaction & mouvement (instantanéité)

- **UI optimiste** à la création : la ligne apparaît en tête du flux immédiatement, en état « enregistrement… » (léger shimmer), confirmée au retour serveur, annulée proprement en cas d'erreur.
- **HTMX partout pour les fragments** : `hx-target` sur le flux du jour, swap localisé, pas de rechargement complet.
- **Flux du jour auto-rafraîchi** : pastille « N nouvelles entrées — actualiser » plutôt qu'un polling agressif ; respecter le réglage utilisateur « Autoriser l'actualisation automatique des widgets » (déjà présent dans Paramètres).
- **Recherche débouncée** : `hx-trigger="keyup changed delay:300ms"`.
- **Clavier :** `Ctrl/⌘ + K` palette · `Échap` ferme le drawer · `Entrée` soumet · navigation `Tab` logique.
- **Mouvement sobre.** Slide-in drawer, fade léger des swaps, point pulsant pour `immediate`. Rien de plus — l'excès d'animation fait « généré » et ralentit l'accueil. Respecter `prefers-reduced-motion`.

## 11. Accessibilité & qualité (plancher non négociable)

- Focus visible (`focus-visible` ring) sur tous les contrôles.
- Drawers : `role="dialog"`, `aria-labelledby`, focus piégé, retour de focus.
- Couleur jamais seule porteuse de sens (icône + libellé + couleur).
- Cibles tactiles ≥ 40px si un poste tablette est utilisé à l'accueil.
- Contraste AA minimum sur texte et badges.
- `aria-live="polite"` sur les toasts et compteurs.

## 12. Performance

- **Partials HTMX** : ne recharger que les fragments (flux, carte routage, ligne mise à jour).
- **Éviter les N+1** dans le flux du jour : `select_related("related_student", "created_by", "branch")` ; `prefetch_related` si besoin.
- **Réutiliser les index existants** : le modèle a déjà de bons index composites (`status`/`created_at`, `branch`/`created_at`, `priority`/`status`…). Aligner les tris/filtres front sur ces index (ex. tri flux = priorité+heure, filtres = statut/service).
- Recherche débouncée + pagination/limite sur les listes longues (116 classes, listes étudiants).
- Charger les polices via `font-display: swap`, sous-ensembles latins.
