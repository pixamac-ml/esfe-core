AUDIT CRITIQUE - DASHBOARD INFORMATICIEN (role "it_support")
================================================================================
Perimetre : `portal/views/it_workflows.py` (1709 lignes), `portal/views/views.py`
(`_render_it_dashboard`, `it_portal`), `portal/services/informaticien_workflows.py`
(350 lignes), `portal/services/notes_workflow.py` (361 lignes),
`portal/selectors/it_workflow_selectors.py`, `portal/views/it_grades_import.py`
(legacy), `portal/urls.py` (~50 routes `it_*`), templates
`templates/portal/informaticien/` (27 fichiers) + legacy
`templates/portal/staff/informaticien_dashboard.html`,
`portal/templates/portal/staff/it_grades_import.html`,
`dashboard_informaticien.html` (racine du repo).

Clarification prealable importante :
- Le role "informaticien" (position="it_support") est mappe vers le role
  canonique `staff_admin` (accounts/access.py ligne 34), et fait partie de
  `positions = {"super_admin", "it_support"}` (ligne 150).
- A la connexion, `portal_dashboard` (portal/views/views.py) detecte
  position=="it_support" et appelle `_render_it_dashboard()` (lignes
  2039-2052), qui construit le contexte via
  `build_home_context()` / `build_it_grade_selection_context()` et rend
  `portal/informaticien/dashboard.html`.
- => CONTRAIREMENT au cas de la secretaire, le routage standard n'est PAS un
  cul-de-sac : `test_portal_dashboard_renders_it_dashboard_from_single_entry`
  (accounts/tests.py) confirme que cette route fonctionne reellement.
- Il existe en plus une route `portal_it` -> `it_portal()` (portal/views/views.py
  lignes 2716-2717) qui ne fait que `redirect("accounts_portal:portal_dashboard")`
  - un stub mort mais inoffensif (voir PROBLEME 7).
- Le dashboard est "100% dynamique cote navigateur" : une coquille Alpine.js
  (`itFlowDashboard()` dans dashboard.html) + ~50 routes HTMX qui chargent des
  "workspaces" dans `#it-workspace`.

CONSTAT GENERAL : ce dashboard est BEAUCOUP plus mature que ne l'etait celui de
la secretaire au depart de son propre audit. La majorite des fonctionnalites
metier critiques (notes, structure academique, comptes, archives, support,
import/export, cartes) sont REELLEMENT implementees, branch-aware, et en
grande partie testees. Les problemes ici ne sont pas "il manque tout", mais
"plusieurs morceaux deja construits ne sont pas branches dans l'interface,
et 2-3 bugs de logique se cachent dans du code par ailleurs solide".

[1.1] Filtrage obligatoire par annexe - respecte dans (presque) tout
------------------------------------------------------------------
`portal/selectors/it_workflow_selectors.py` centralise l'acces aux classes,
semestres et etudiants : `get_it_academic_classes()`, `get_it_semesters_for_class()`,
`get_it_students_for_class()` filtrent systematiquement par
`academic_class__branch=branch` (et excluent les classes archivees). Le branch
est resolu via `get_user_branch(request.user)` (accounts/access.py).

`portal/services/informaticien_workflows.py` applique la meme regle a tout le
reste : `build_home_context`, `build_support_context`, `build_audit_context`,
`build_import_context`, `build_structure_context`, `build_supervision_context`
prennent tous un `branch` en parametre et filtrent leurs querysets dessus
(`AccountSupportState.objects... filter(user__profile__branch=branch)`,
`SupportAuditLog... filter(branch=branch)`, etc.).

`it_workflows.py` ajoute une seconde ligne de defense pour les actions cote
formulaire : `_same_branch_or_forbidden()` (lignes 84-89) verifie explicitement
que l'objet cible (classe, etudiant, compte) appartient bien a la meme annexe
que l'utilisateur avant toute action de modification - sinon HttpResponseForbidden.
=> Teste explicitement : `test_it_cross_branch_access_rejected` (accounts/tests.py
ligne ~2063).

EXCEPTION CONNUE : le catalogue (Language/Profession) n'est PAS filtre par
annexe - c'est volontaire dans le code actuel (donnees globales/referentiels),
mais merite discussion (voir PROBLEME 5).

[1.2] Machine a etats des notes - `portal/services/notes_workflow.py`
------------------------------------------------------------------
Vrai etat machine, pas une simulation :
- Etats : STATE_EMPTY -> STATE_IN_PROGRESS -> STATE_READY_TO_PUBLISH_NORMAL ->
  STATE_NORMAL_PUBLISHED -> STATE_RETAKE_IN_PROGRESS ->
  STATE_READY_TO_PUBLISH_FINAL -> STATE_FINAL_PUBLISHED, pilotes par
  `Semester.status` (STATUS_DRAFT/STATUS_NORMAL_ENTRY/STATUS_NORMAL_LOCKED/
  STATUS_RETAKE_ENTRY/STATUS_FINALIZED/STATUS_PUBLISHED).
- `get_notes_state()` calcule l'etat reel a partir des notes saisies
  (notes manquantes, notes en attente de calcul, etc.) - pas un statut
  declaratif.
- `get_available_actions()` retourne dynamiquement les actions valides pour
  l'etat courant (verrouiller, publier normal, ouvrir rattrapage, publier
  final...).
- `apply_notes_workflow_action()` applique l'action, journalise dans
  `SupportAuditLog` via `log_support_action()`, et est appele par
  `it_notes_workflow_action()` (it_workflows.py lignes 349-397).
- `get_retake_candidates()` / `can_edit_retake_grade()` : seuls les etudiants
  effectivement eligibles au rattrapage (note < seuil) peuvent voir leur note
  de rattrapage editee - empeche de "forcer" un resultat, conformement a
  `reponse_role_informaticien.md` ("Il ne doit pas avoir de bouton 'Forcer
  passage'").

[1.3] KPIs reels par classe/semestre - `it_notes_kpi`
------------------------------------------------------------------
`it_notes_kpi()` (it_workflows.py lignes 139-214) calcule reellement :
effectif, notes saisies/attendues, notes validees, notes en echec,
dettes academiques en attente, pourcentage de progression - tout est requete
sur les modeles reels (pas de chiffres en dur). Rendu dans
`portal/informaticien/partials/notes_kpi.html`, branche via hx-get dans
`ui/templates/notes/notes_workflow.html` (hx-trigger="load, kpi-update from:body").

[1.4] Decision annuelle officielle - `academics/services/year.py`
------------------------------------------------------------------
`compute_annual_decision(enrollment)` (lignes 231-322) implemente la VRAIE
logique : VALIDE / ADMISSIBLE / NON_ADMIS, en s'appuyant sur
`AcademicClass.admissibility_gap` (defaut 2.00), et cree les
`AcademicDebt` via `create_academic_debts()` quand ADMISSIBLE. C'est cette
fonction qui devrait etre la SEULE source de verite pour les decisions
annuelles (voir PROBLEME 2 - `it_notes_decisions` ne l'utilise pas).

- `it_supervision_workspace` (ligne 342) -> `supervision_workspace.html`

Aucune des trois n'apparait dans `modules`/`openModule`, ni dans aucun autre
template sous `templates/portal/informaticien/` (verifie par grep global).
=> Ce sont des fonctionnalites COMPLETES (logique + template + branch-scoping)
qui n'ont jamais ete reliees a l'UI. C'est exactement le type de "trou de
navigation" que l'audit de la secretaire avait identifie comme prioritaire

=> IMPACT : `it_supervision_workspace` est en particulier celui qui correspond

PROBLEME 2 : `it_notes_decisions` reimplemente (mal) la logique de decision annuelle
------------------------------------------------------------------------------
`it_workflows.py` lignes 241-316 (`it_notes_decisions`) recalcule "a la main"
les decisions par semestre et la decision annuelle, AU LIEU d'appeler
`compute_annual_decision()` (academics/services/year.py lignes 231-322), qui
est la fonction officielle utilisee ailleurs dans le projet (et qui gere
correctement VALIDE / ADMISSIBLE / NON_ADMIS + creation des `AcademicDebt`).

Consequence concrete : dans `it_notes_decisions`, chaque decision de semestre
ne peut valoir que "VALIDE" ou "NON ADMIS" (jamais "ADMISSIBLE"), donc la
branche `elif s1_ok and s2_ok: year_decision = "ADMISSIBLE"` est du CODE MORT
- elle ne peut structurellement jamais s'executer. Un etudiant qui devrait
etre "ADMISSIBLE" (avec creation de dette academique via
`create_academic_debts()`) sera donc classe soit VALIDE soit NON_ADMIS par
cette vue, ce qui DIVERGE du calcul officiel utilise par ailleurs (bulletins,
portail etudiant, etc.).

De plus, la route `it_notes_decisions` (portal/urls.py ligne 316) n'est
appelee depuis AUCUN template du dashboard informaticien (seulement
referencee dans `test_urls.py`) - c'est donc a la fois un bug de logique ET
une fonctionnalite non branchee a l'UI.

PROBLEME 3 : bug "ready_to_calculate" - la section "Resultats" de l'accueil est morte
------------------------------------------------------------------------------
`portal/services/informaticien_workflows.py` ligne 86, dans
`build_home_context()` :

    if state.code == "ready_to_calculate":
        calculable_classes.append(item)

Or `notes_workflow.py` ne definit AUCUN etat avec le code
`"ready_to_calculate"`. Les codes reels sont : `empty`, `in_progress`,
`ready_to_publish_normal`, `normal_published`, `retake_in_progress`,
`ready_to_publish_final`, `final_published`.

=> `calculable_classes` est donc TOUJOURS une liste vide, quel que soit l'etat
reel des classes. Dans `home_workspace.html`, le bloc "Resultats - pretes a
calculer" (avec le bouton "Generer resultats") affiche donc systematiquement
"Aucune classe prete a calculer", MEME quand une classe a un semestre
effectivement pret a etre publie (`ready_to_publish_normal`/
`ready_to_publish_final`).
=> Petite cause (un seul comparateur de chaine), gros effet visible (une
section entiere de l'accueil semble "ne jamais avoir rien a faire").
Note : la classe CSS `.workflow-band-ready_to_calculate` existe aussi
(dashboard.html ligne 42) et est donc elle aussi morte.

9.1 [RESOLU - voir section 8] LE PLUS IMPORTANT : les "decisions de fin
    d'annee" des etudiants etaient calculees DEUX FOIS, avec DEUX RESULTATS
    DIFFERENTS
--------------------------------------------------------------------------

Corrige (Option A appliquee) : `it_notes_decisions` appelle desormais
`compute_annual_decision()`, exactement comme les bulletins. Le cas
"ADMISSIBLE" peut maintenant apparaitre dans cet ecran. Detail complet en
section 8, entree "[FAIT] PROBLEME 2 / 9.1".

Description du probleme d'origine (laissee pour memoire) :

Concretement :
- Quand le systeme decide si un etudiant est "VALIDE" (passe en annee
  superieure), "ADMISSIBLE" (passe mais avec une ou deux matieres a repasser
  / dette academique) ou "NON ADMIS" (redouble), il existe UNE SEULE bonne
  facon officielle de calculer ca : la fonction `compute_annual_decision()`.
  C'est elle qui est utilisee pour les bulletins et le portail etudiant.

- Mais l'ecran "it_notes_decisions" du dashboard informaticien (pas encore
  visible dans le menu, donc personne ne le voit pour l'instant) refait ce
  calcul A LA MAIN, en moins complet. Resultat : dans CET ecran-la, un
  etudiant qui devrait etre "ADMISSIBLE" (rattrapage/dette) sera affiche soit
  "VALIDE" soit "NON ADMIS" - jamais "ADMISSIBLE". Le cas "ADMISSIBLE"
  n'arrive JAMAIS dans cet ecran, alors qu'il arrive normalement dans les
  bulletins officiels.

Pourquoi c'est le plus important :
- Tant que cet ecran reste cache (pas dans le menu), ce n'est PAS dangereux
  - personne ne s'en sert.
- MAIS si un jour quelqu'un l'ajoute au menu (ou s'en sert pour annoncer des
  resultats), il y aura un ecran qui dit "Karim = NON ADMIS" alors que son
  bulletin officiel dit "Karim = ADMISSIBLE (rattrapage)". Deux sources de
  verite differentes = confusion garantie devant un directeur ou un parent.

Ce qu'on propose :
- Option A (la bonne, prend un peu plus de temps) : reecrire cet ecran pour
  qu'il appelle la MEME fonction officielle `compute_annual_decision()` que
  les bulletins. Un seul calcul, partout les memes resultats.
- Option B (rapide, en attendant) : ne rien casser, mais ajouter une note/
  flag clair dans le code pour dire "ne pas brancher cet ecran au menu tant
  que l'option A n'est pas faite" - pour eviter qu'on l'active par erreur
  plus tard.

9.2 Le "catalogue" (langues, professions) est visible/modifiable par TOUTES
    les annexes
--------------------------------------------------------------------------

Concretement :

  comme partout ailleurs.

Ce qu'on propose :

  dans l'ecran pour que ce soit clair, et on ferme le sujet (5 minutes).
- Si "chaque annexe a sa liste" -> travail un peu plus consequent (modele de
  donnees + filtrage), a planifier a part.
