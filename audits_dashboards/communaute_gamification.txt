PROBLEMES IDENTIFIES ET CORRIGES
----------------------------------------------------------------

1. Classement (leaderboard.html) - bug d'affichage
   - Probleme : le template utilisait {{ entry.answers_count }} et
     {{ entry.accepted_count }}, des champs qui n'existent PAS sur
     GamificationProfile -> rendu silencieusement vide (Django n'affiche
     rien sur un attribut inconnu, donc aucune erreur visible mais colonnes
     toujours a blanc).
   - Correction : remplace par les vrais champs answers_given et
     answers_accepted.

2. Onglet "Badges" du profil - donnees fictives
   - Probleme : partials/profile/badges.html affichait des badges
     codes en dur ("Debutant / Contributeur / Expert / Specialiste")
     avec une fausse progression "3/10", deconnectes de la vraie BDD
     de badges (BadgeDefinition / UserBadge).
   - Correction : reecriture complete du template :
     - en-tete niveau/XP (icone de niveau, titre, total_xp, barre de
       progression xp_profile)
     - grille des badges reellement obtenus (icone, nom, description,
       rarete, date d'obtention)
     - nouvelle section "A debloquer" : calcule dans profile_badges()
       (views.py) les 8 prochains badges non obtenus avec la progression
       reelle vers la premiere condition (answers_count, accepted_answers,
       upvotes_received, topics_count, streak_days, level)
     - etat vide explicite si l'utilisateur n'a encore aucun badge

3. Badges/niveau invisibles sur le forum (PROBLEME LE PLUS IMPORTANT)
   - Probleme racine double :
     a) le widget level_badge.html (icone + niveau a cote du pseudo)
        n'etait inclus dans aucun template de reponse.
     b) topic_detail.html (page principale d'un sujet) ne rendait PAS le
        partial answer_item.html / reply_item.html pour la liste des
        reponses : il avait sa propre boucle minimaliste (auteur + texte
        seulement, sans votes, sans bouton "accepter", sans sous-reponses,
        sans badge de niveau). Resultat : meme apres ajout du widget dans
        answer_item.html, RIEN ne serait visible au chargement normal
        d'une page de sujet (seulement sur une reponse ajoutee en direct
        via HTMX pendant la session).
   - Correction :
     - ajout de {% include ".../level_badge.html" with user=answer.author %}
       dans answer_item.html ET reply_item.html
     - topic_detail.html : la boucle "Reponses" rend maintenant
       answer_item.html pour chaque reponse principale (votes, bouton
       "accepter", sous-reponses, badge de niveau inclus)
     - vues.py : ajout de author__gamification aux select_related
       (reponses principales + sous-reponses prefetchees) pour eviter le
       N+1 du nouveau widget
     - correction d'un bug annexe decouvert au passage : answer.is_accepted
       n'existe pas comme champ -> {% if answer.is_accepted %} etait
       TOUJOURS faux (le surlignage "Reponse validee" ne s'affichait
       jamais). Calcul desormais fait dans la vue
       (answer.is_accepted = answer.id == topic.accepted_answer_id)
       et transmis au partial.

4. Trois actions XP "mortes" (jamais declenchees)
   - daily_login + series (streak) :
     bug dans update_daily_streak() qui retournait profile.current_streak
     (un entier toujours > 0 apres la 1ere connexion -> aurait donne les XP
     de connexion quotidienne a CHAQUE connexion, pas une fois par jour).
     Corrige pour retourner le booleen "premiere activite du jour".
   - first_topic_of_day (+20 XP au 1er sujet du jour) : signal ajoute.
   - complete_profile (+30 XP, une seule fois, quand avatar + bio remplis) :
     signal ajoute, avec verification anti double-attribution via
     XPTransaction.
   - Verifie de bout en bout (script de test temporaire, supprime apres) :
     connexion 1 -> +2 XP "daily_login" ; connexion 2 (meme jour) -> pas de
     doublon ; bio seule -> 0 XP ; avatar+bio -> +30 XP ; re-sauvegarde ->
     pas de doublon ; 1er sujet du jour -> +15 (creation) +20 (bonus) = 35 ;
     2e sujet du jour -> +15 seulement.

5. Pas de feedback visuel a l'obtention d'XP
   - Ajout de notifications "toast" (+X XP) via swap HTMX OOB
     (hx-swap-oob="afterbegin:#xp-toast-container"), animation Alpine.js
     (apparition/disparition automatique apres 3s).
   - Declenchees sur : creation de reponse (create_answer) et upvote donne
     (give_upvote, uniquement pour un NOUVEAU vote positif).
   - Conteneur #xp-toast-container ajoute dans topic_detail.html.

6. Coherence transactionnelle
   - award_xp() (gamification.py) enveloppe desormais l'ajout d'XP, la mise
     a jour des stats (F() expressions) et la verification des badges dans
     un transaction.atomic().

================================================================
PARTIE 2 - VISION & PLAN D'AMELIORATION (avant demo DG lundi)
================================================================

LA VISION (telle que comprise) :
La communaute ESFE doit fonctionner comme un "Stack Overflow" sante au
Mali : etudiants ET enseignants experimentes echangent sur leurs themes /
problemes quotidiens. La gamification (XP, niveaux, badges, classement)
sert a faire emerger NATURELLEMENT les profils les plus actifs/utiles
(notamment les enseignants qui repondent regulierement) et a les "mettre
en avant" via le systeme de profil, le classement et les widgets de la
page communaute. C'est un argument fort pour le DG : le site cree de la
visibilite/reconnaissance pour les enseignants ET un outil d'entraide pour
les etudiants, sans equivalent "sante" au Mali.
