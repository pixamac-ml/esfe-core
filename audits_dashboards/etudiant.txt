================================================================
Fichiers principaux :
- templates/portal/student/dashboard.html (page-cadre, Alpine.js "studentDashboard()")
- templates/portal/student/partials/*.html (9 sections + sous-panneaux)
- static/portal/student/dashboard.js (~650 lignes, logique Alpine + lazy-load HTMX)
- static/portal/student/dashboard.css (theme, ~1234 lignes)
- static/portal/student/course_immersive.js (lecteur de cours "Academy")

- portal/student/widgets/academics.py, finance.py, profile.py
- portal/student/profile_service.py (centre de gestion du compte)

Acces : /portal/student-dashboard/. Filtrage naturel par etudiant connecte
(get_student_academic_snapshot(user) -> student/enrollment/class/programme/
annee/niveau/ECs/statut). Pas de notion d'"annexe transverse" ici puisque
chaque etudiant ne voit que SES propres donnees -> pas de risque de fuite
inter-annexe sur ce dashboard (contrairement au DE/Gestionnaire).

----------------------------------------------------------------
0. ARCHITECTURE GENERALE
----------------------------------------------------------------
- dashboard.html est un shell Alpine (x-data="studentDashboard()") avec
  9 sections en x-show : overview, courses, schedule, messages, academics,
  teachers, diplomas, shop, settings.
- Chaque section (sauf overview) est chargee en lazy-load via HTMX la
  premiere fois qu'on y navigue : attribut data-section-url +
  data-section-loaded="0", puis ensureSectionLoaded(sectionKey) declenche
  htmx.ajax(...) et bascule data-section-loaded a "1".
- ✅ Le theme CSS (--school-primary: #1e4f6f, --school-secondary: #1db5b0,
  --school-accent: #f39c12, palette dans dashboard.css :root) correspond
  REELLEMENT a la palette compilee du projet (tailwind.config.js : primary
  DEFAULT #1e4f6f, secondary #1db5b0, accent #f39c12). C'est un GROS point
  positif compare au dashboard DE (qui charge Tailwind CDN avec une palette
  bleu marine totalement differente). Le dashboard etudiant est visuellement
  "raccorde" au reste du site.
- ✅ Mode sombre fonctionnel : classe .dark-preview sur student-dashboard-page
  qui re-definit toutes les variables CSS (surface/card/text/muted/...). Pas
  un gadget cosmetique : ca couvre vraiment toutes les surfaces (topbar,
  cartes, texte, bordures).
- ✅ Responsive serieux : breakpoints 1199px / 991px / 640px qui reorganisent
  la sidebar (off-canvas mobile), la grille academy, le lecteur de cours, etc.

1. ACCUEIL (section "overview")
----------------------------------------------------------------
- hero_student.html : salutation, mini-calendrier (mois courant, navigation
  prev/next), carte "Prochain cours" (next_course), grille de
  "priority_actions" (jusqu'a 4 cartes).
- get_student_priority_actions(student, snapshot, next_course, stats) genere
  des actions VRAIMENT dynamiques et utiles :

  - completion du profil (% + lien vers Parametres si incomplet)

  - reste a payer (lien vers Finance, si > 0)

  l'etat REEL de l'etudiant (pas de texte statique).
- overview_schedule_quick.html : apercu du planning de la semaine (4
  evenements max), bouton "Voir tout" -> goToSection('schedule', ...).

  est mort. A confirmer si un futur dev voulait l'integrer dans l'accueil
  (ca donnerait une vue "progression globale" en plus des priority actions).

2. MES COURS (section "courses" - vue "Academy")
----------------------------------------------------------------
- courses_student.html (242 lignes) : catalogue style "academy" avec filtres
  Alpine (semestre, statut, recherche texte), pagination cote client,
  cartes de cours (academy-cover-1..4 selon index, badges "verrouille"/
  "obligatoire", barre de progression).
- Chaque carte a 2 actions :
  - "Apercu" -> openCoursePreview(ec_preview_url, titre) (si semestre

  - "Ouvrir" -> openCourseDetail(ec_detail_url, titre) (si semestre

  is_semester_unlocked_for_enrollment(enrollment, semester_number) (cote

  verrouille (403, pas juste un cache visuel).
- ✅ _apply_course_semester_unlocks (services.py) calcule cet etat une
  seule fois pour toute la liste -> pas de N requetes redondantes par carte.

3. LECTEUR DE COURS "ACADEMY" (overlay ec_detail / ec_preview)
----------------------------------------------------------------
- openCourseDetail() charge ec_detail (avec ?partial=1 -> rend
  partials/ec_detail_panel.html) dans #studentCourseDetailBody, en plein
  ecran (student-detail-shell.is-course-reader).
- ec_detail_panel.html (135 lignes) : vrai lecteur "Academy" avec
  Alpine "studentCourseImmersive()" (course_immersive.js) :
  - sidebar "Playlist" : chapitres -> lecons, avec icone check pour les
    lecons terminees, % de progression par lecon, barre de progression
    globale (total_completed/total_active).
  - zone principale : navigation Precedent/Suivant entre lecons
    (prevLesson/nextLesson, bases sur l'ordre du player_bundle JSON),
    affichage du titre de la lecon active.
  - ec_content_lesson.html (103 lignes) gere TOUS les types de contenu :
    video (fichier local OU embed externe OU URL directe), audio, image,
    PDF (iframe), texte (whitespace-pre-line), document bureau (avec
    fallback Office Online embed ou bouton "Ouvrir dans le lecteur" /
    "Telecharger"). Bouton "Marquer termine" (data-complete-content) +
    tracking auto de progression (data-track-content / data-media-progress-
    content) pour video/audio/iframe/texte/image.
  - "Lecteur integre" (docViewerUrl) : overlay plein ecran pour ouvrir un
    fichier non-previsualisable dans un iframe.
  ✅ C'est de loin la fonctionnalite la plus "produit fini" de tout
  l'audit jusqu'ici : gestion fine des cas (chapitres vides, contenu non
  publie, contenu sans ressource, etc.) avec des messages explicites a
  chaque etat.
- ec_preview.html (123 lignes) : version "allegee" affichee dans
  openCoursePreview() (modale plus petite, is-course-preview). Choisit un
  preview_content (priorite : has_inline_preview, sinon
  has_accessible_source). Gere les memes types de contenu en plus compact,
  avec lien "Ouvrir" -> openCourseDetail (bascule vers le lecteur complet).
  ✅ Bonne separation "apercu rapide" vs "lecteur complet", logique view
  (ec_preview) tres proche de ec_detail (memes controles d'acces /
  verrouillage de semestre).
- 🚩 Encodage : dans dashboard.js, performGlobalSearch() construit un
  sous-titre de resultat avec un caractere mal encode :
    `Semestre ${course.semester} Â· ${course.code}`
  -> "Â·" au lieu de "·" (point median). Visible dans le panneau de
  recherche globale (icone loupe en haut du dashboard) quand on tape un
  nom de cours. Petit defaut visuel mais facilement repere si le DG fait
  une recherche en live.
