PROPOSITION DE PRIX - PROJET ESFE
Date d'analyse : 25 mai 2026

Contexte personnel :
Tu es deja employe dans l'ecole comme Responsable marketing digital. Cela change la maniere de negocier : il ne faut pas presenter ce projet comme un simple site web, ni comme une petite tache comprise automatiquement dans ton poste. Le projet ressemble a un systeme integre de gestion scolaire, administrative, financiere et pedagogique.


1. Lecture rapide du projet
---------------------------
Apres analyse du depot, le projet ESFE contient :

- un site public institutionnel : accueil, a propos, formations, blog, actualites, contact, pages legales, SEO, sitemap, pages d'erreur ;
- un module admissions : candidature, dossiers, statuts, pieces jointes ;
- un module inscriptions : transformation des candidatures validees en inscriptions administratives ;
- un module paiements : especes, mobile money, virement, validation, recus, agents, sessions de caisse, corrections, logs financiers ;
- un module etudiants : creation automatique apres paiement, matricule, profil, lien avec l'inscription ;
- un module academique : annees academiques, classes, UE/EC, contenus, notes, resultats, bulletins, diplomes, emplois du temps ;
- un portail etudiant ;
- des portails staff : direction generale, direction des etudes, surveillant academique, enseignant, finance, admissions, secretariat, informaticien/support IT ;
- un superadmin transversal ;
- un module secretariat : rendez-vous, visiteurs, registre, taches, documents ;
- un module communaute avec notifications, badges, reputation/gamification ;
- un module boutique/shop ;
- une architecture Django avancee avec PostgreSQL, Redis/WebSockets possibles, HTMX, Tailwind, PDF, QR code, emails et tests automatises.

Quelques indicateurs du depot :
- environ 688 fichiers Python ;
- environ 594 templates HTML ;
- plus de 100 modeles metier Django ;
- une base de tests significative ;
- verification Django executee avec succes : manage.py check = aucune erreur detectee.

Conclusion : ce projet ne doit pas etre vendu au prix d'un site vitrine. Il s'agit plutot d'un ERP scolaire web adapte a l'ESFE.


2. Valeur estimee du projet
---------------------------
Pour le marche malien, il faut rester realiste, mais ne pas sous-estimer le travail. Une plateforme de ce niveau peut etre valorisee en plusieurs manieres.

Option A - Prix interne minimal acceptable
Prix conseille : 3 000 000 a 5 000 000 FCFA

Cette option correspond a une reconnaissance interne du travail deja fourni, sans facturer au prix complet d'une societe logicielle. C'est le minimum serieux si l'ecole veut utiliser le systeme comme outil officiel.

Je ne recommande pas de descendre sous 3 000 000 FCFA, sauf si l'ecole signe en contrepartie un contrat clair de prime, reconnaissance de propriete intellectuelle, evolution de ton poste et facturation separee des futures versions.


Option B - Prix juste et defendable pour l'ecole
Prix conseille : 6 000 000 a 10 000 000 FCFA

C'est la fourchette que je recommande pour la negociation principale.

Elle reste raisonnable pour une ecole au Mali, mais elle reconnait que le projet couvre plusieurs services : marketing, admissions, finance, scolarite, pedagogie, direction, secretariat et support IT.

Proposition concrete :
- 7 500 000 FCFA pour la livraison du systeme ;
- 1 500 000 a 2 500 000 FCFA pour l'industrialisation production, si elle n'est pas encore finalisee ;
- evolutions futures facturees par version ou par lot fonctionnel, et non comme maintenance mensuelle.


Option C - Prix agence / valeur complete du logiciel
Prix estime : 12 000 000 a 20 000 000 FCFA ou plus

Si l'ecole devait commander ce projet a une agence ou a une equipe externe, le cout pourrait facilement depasser cette fourchette, surtout avec les modules paiements, notes, bulletins, portails, workflows multi-roles et mise en production.

Cette option sert surtout d'argument comparatif : elle permet de montrer que ton prix interne reste avantageux pour l'etablissement.

4. Ce qu'il faut separer du prix principal
-----------------------------------------
Le prix principal ne doit pas tout inclure sans limite. Il faut separer :

1. Livraison du logiciel existant
   - code source ;
   - configuration ;
   - base fonctionnelle ;
   - documentation minimale ;
   - formation initiale des utilisateurs cles.

2. Industrialisation production
   - VPS ou serveur ;
   - PostgreSQL production ;
   - Redis si WebSockets actifs ;
   - nom de domaine ;
   - SSL ;
   - sauvegardes ;
   - emails SMTP ;
   - securisation ;
   - tests de parcours critiques.

Budget conseille : 1 500 000 a 2 500 000 FCFA.

3. Cycle de versions et evolutions
   L'idee n'est pas de parler de maintenance mensuelle. Il est plus strategique de presenter le projet comme un logiciel vivant avec des versions :

   - V1.0.0 : premiere version officielle mise en production ;
   - V1.1.0, V1.2.0 : versions mineures avec ameliorations ciblees ;
   - V2.0.0 : version majeure avec refonte, nouveau module important ou changement profond de processus.

   Pendant l'annee, les besoins peuvent etre observes, notes et prepares. La publication principale des ameliorations peut etre programmee pendant les vacances, pour eviter de perturber l'annee academique.

   Tarification conseillee :
   - version mineure : 150 000 a 200 000 FCFA selon le volume ;
   - version majeure : a partir de 300 000 FCFA, a ajuster selon la refonte ou les modules ajoutes ;
   - tres gros module ou refonte complete : devis separe.

Cette approche est plus facile a faire accepter, car l'ecole paie pour une evolution visible et planifiee, pas pour une charge mensuelle abstraite.

4. Evolutions majeures
   A facturer separement : nouveaux modules, refonte UI, application mobile, paiement mobile automatise, integration API externe, rapports statistiques avances, migration massive de donnees historiques.


5. Strategie adaptee a ta position d'employe
--------------------------------------------
Comme tu es deja Responsable marketing digital, il faut eviter une confrontation du type "je vous vends le logiciel comme un prestataire externe". La meilleure approche est :

- parler de "valorisation exceptionnelle du projet" ou "prime projet numerique" ;
- demander un document ecrit qui reconnait ton role ;
- clarifier ce qui appartient a l'ecole et ce qui reste ton savoir-faire ;
- obtenir une logique de versions facturees ou une augmentation de responsabilite ;
- ne pas laisser la direction croire que tout est inclus dans ton salaire actuel.

Proposition de structure :

Paiement initial :
- 50% a la validation du projet ;
- 30% a la mise en ligne ;
- 20% apres recette fonctionnelle.

Ou, si l'ecole prefere lisser :
- 3 000 000 FCFA maintenant ;
- 2 000 000 FCFA a la mise en production ;
- 500 000 a 1 000 000 FCFA par mois pendant 6 a 10 mois selon l'accord ;
- evolutions facturees ensuite par version, surtout avant les vacances.

Strategie de versioning conseillee :

- V1.0.0 : livraison officielle de la plateforme actuelle stabilisee.
- V1.1.0 : petites ameliorations d'interface, corrections mineures, ajustements de workflow.
- V1.2.0 : nouveaux rapports, meilleurs filtres, ameliorations des tableaux de bord.
- V2.0.0 : gros module, refonte importante, automatisation avancee ou nouvelle architecture fonctionnelle.

Phrase utile :

"Je ne propose pas une maintenance mensuelle obligatoire. Je propose de faire evoluer le logiciel par versions. La V1 sera livree officiellement, puis les ameliorations seront regroupees et publiees a des moments adaptes, surtout pendant les vacances, avec un cout selon le niveau de changement."


6. Points de vigilance avant production
---------------------------------------
Le projet est avance, mais avant de le presenter comme pret a 100% pour la production, il faut prevoir :

- audit de securite : DEBUG=False, SECRET_KEY, ALLOWED_HOSTS, CSRF, cookies securises ;
- serveur de production propre : VPS, PostgreSQL, Redis, Nginx, SSL ;
- plan de sauvegarde automatique ;
- tests des parcours critiques : candidature, inscription, paiement, creation etudiant, notes, bulletins, roles utilisateurs ;
- nettoyage des donnees de demonstration ;
- documentation utilisateur ;
- formation des agents ;
- clarification des responsabilites : qui valide les paiements, qui corrige les notes, qui gere les utilisateurs, qui restaure les sauvegardes.

9. Si la DG demande des factures pour l'hebergement
---------------------------------------------------
Il faut etre tres prudent. L'hebergement n'est pas ton revenu personnel : c'est un frais externe facture par un fournisseur comme Hostinger, OVH, Namecheap, Google, Microsoft, etc.

Reponse conseillee :

"Pour l'hebergement et le nom de domaine, je recommande que l'ecole paie directement le fournisseur avec ses propres informations, afin que les factures soient au nom de l'ecole. Moi, je peux accompagner le choix, la configuration et le deploiement technique."

Pourquoi c'est mieux :

- la facture est directement au nom de l'ecole ;
- tu n'es pas confondu avec le fournisseur d'hebergement ;
- l'ecole garde la propriete du domaine et du serveur ;
- cela evite les accusations du type "il a ajoute une marge cachee" ;
- c'est plus professionnel pour la comptabilite.

Si l'ecole veut que tu avances les frais :

"C'est possible, mais dans ce cas les frais d'hebergement et de domaine seront refactures exactement selon les justificatifs du fournisseur, separement du prix du logiciel."

Dans ce cas, il faut fournir :

- la facture ou le recu du fournisseur ;
- le montant exact ;
- la duree couverte ;
- le service achete ;
- le compte ou domaine concerne.

Phrase a eviter :

"L'hebergement est inclus dans les 10 millions."

Pourquoi l'eviter :

- si le serveur coute plus cher plus tard, tu portes le risque ;
- si l'ecole veut une facture fournisseur, tu seras bloque ;
- si le DG pense que tout est inclus pour plusieurs annees, tu perds la maitrise du perimetre ;
- si le trafic augmente, les couts serveur peuvent changer.

--------------------------------
Il faut presenter trois blocs separes :

Bloc 1 - Valorisation du logiciel V1.0.0
Montant : 10 000 000 FCFA annonce, 7 500 000 FCFA cible, 5 000 000 FCFA minimum.

Bloc 2 - Frais techniques externes
Exemples : hebergement, nom de domaine, email professionnel, SMS, passerelle de paiement.
Ces frais doivent etre payes directement par l'ecole ou rembourses sur facture fournisseur.

Bloc 3 - Evolutions futures par version
Version mineure : 150 000 a 200 000 FCFA.
Version majeure : a partir de 300 000 FCFA.
Gros module/refonte : devis specifique.

Phrase de negociation :

"Pour que les choses soient transparentes, je separe le prix du logiciel des frais externes. Le montant de 10 000 000 FCFA concerne la V1.0.0 du systeme ESFE. L'hebergement, le domaine et les services externes seront payes directement par l'ecole ou rembourses sur justificatif. Ensuite, les ameliorations seront gerees par versions."
