
DASHBOARD GESTIONNAIRE ESFE - EVOLUTION METIER ET AUTOMATISATION

Date : 2026-05-29

OBJECTIF GENERAL

Le dashboard Gestionnaire doit devenir le véritable centre de pilotage opérationnel et financier d'une annexe ESFE.

La gestionnaire ne doit plus passer son temps à préparer manuellement les calculs, les fiches de paiement ou les consolidations financières.

Le système doit automatiser la préparation des données.
La gestionnaire doit contrôler, corriger si nécessaire, valider et piloter.

====================================================================
1. VISION METIER REELLE DE LA GESTIONNAIRE
====================================================================

Dans une annexe ESFE, la gestionnaire est responsable :

- des candidatures ;
- des inscriptions ;
- des paiements ;
- des entrées de caisse ;
- des sorties de caisse ;
- des dépenses ;
- des salaires du personnel administratif ;
- des honoraires des enseignants ;
- des ventes réalisées dans la boutique ;
- des rapports financiers ;
- de la clôture mensuelle ;
- du versement bancaire du résultat de l'annexe.

Le dashboard doit refléter exactement cette réalité.

====================================================================
2. FILTRAGE OBLIGATOIRE PAR ANNEXE
====================================================================

Toutes les données doivent être filtrées par annexe.

La gestionnaire ne doit jamais voir :

- les candidatures d'une autre annexe ;
- les inscriptions d'une autre annexe ;
- les paiements d'une autre annexe ;
- les salaires d'une autre annexe ;
- les honoraires d'une autre annexe ;
- les dépenses d'une autre annexe ;
- les commandes d'une autre annexe ;
- les rapports d'une autre annexe.

Cette règle est obligatoire et ne doit jamais être cassée.

====================================================================
3. AUTOMATISATION DES SALAIRES DU STAFF
====================================================================

Le système ne doit plus demander à la gestionnaire de créer manuellement les fiches de paie.

Lorsqu'un membre du personnel administratif est créé :

- Informaticien
- Directeur des études
- Surveillant
- Secrétaire
- Secrétaire adjointe
- Gestionnaire
- Gardien
- Autre membre du staff

Le formulaire doit contenir :

Salaire

Ne pas utiliser "salaire de base".

Le salaire enregistré représente le salaire officiel actuel du collaborateur.

Le système doit automatiquement :

- détecter les employés actifs ;
- générer les fiches de paie du mois ;
- prendre en compte les avances ;
- préparer les paiements ;
- notifier la gestionnaire.

La gestionnaire conserve la possibilité :

- d'éditer la fiche ;
- de corriger un montant ;
- d'ajouter un ajustement ;
- de valider ;
- de payer.

====================================================================
4. AUTOMATISATION DES HONORAIRES ENSEIGNANTS
====================================================================

Les enseignants ne doivent pas utiliser le système de salaire.

Ils doivent utiliser un moteur d'honoraires indépendant.

Lors de la création d'un enseignant :

ajouter :

Tarif horaire

Exemple :

5000 FCFA / heure
10000 FCFA / heure

Le système doit automatiquement :

- récupérer les cours réalisés ;
- récupérer les heures validées ;
- calculer les honoraires ;

Formule :

Heures validées × Tarif horaire

Exemple :

12 heures validées
5000 FCFA / heure

Honoraires = 60 000 FCFA

Le système prépare automatiquement la fiche.

La gestionnaire :

- consulte ;
- corrige si nécessaire ;
- valide ;
- paie.

Chaque paiement doit générer :

- reçu ;
- historique ;
- mouvement financier ;
- traçabilité.

====================================================================
5. RAPPORTS DETAILLES
====================================================================

Les rapports doivent devenir beaucoup plus riches.

Rapports disponibles :

- journalier ;
- hebdomadaire ;
- mensuel ;
- période personnalisée.

Les rapports doivent afficher clairement :

ADMISSIONS

- candidatures ;
- acceptations ;
- refus ;
- dossiers à compléter ;
- inscriptions créées.

FINANCES

- paiements étudiants ;
- dons ;
- ventes boutique ;
- autres recettes.

CHARGES

- salaires ;
- honoraires ;
- dépenses ;
- loyer ;
- charges diverses.

RESULTAT

- total entrées ;
- total sorties ;
- résultat net.

Le format doit être proche d'un tableau Excel pour faciliter la lecture.

====================================================================
6. CLOTURE MENSUELLE
====================================================================

La clôture mensuelle est une étape métier obligatoire.

Workflow :

1. Vérification des opérations
2. Vérification des paiements
3. Vérification des salaires
4. Vérification des honoraires
5. Vérification des dépenses
6. Génération du bilan
7. Validation gestionnaire
8. Versement bancaire
9. Clôture du mois

Le système doit archiver la période.

Les données ne doivent jamais être supprimées.

====================================================================
7. VERSEMENTS BANCAIRES
====================================================================

Après clôture :

Le système calcule automatiquement :

Total recettes
- Total charges
= Résultat de période

Le montant restant peut être versé à la banque.

Informations enregistrées :

- banque ;
- référence ;
- date ;
- montant ;
- justificatif ;
- commentaire.

Le DG doit pouvoir consulter les versements reçus depuis les annexes.

====================================================================
8. ARCHIVES ET HISTORIQUE
====================================================================

Chaque mois clôturé doit être archivé.

La gestionnaire doit pouvoir :

- consulter les anciens mois ;
- consulter les anciens rapports ;
- consulter les anciens versements ;
- consulter les anciennes dépenses ;
- consulter les anciens paiements.

Les indicateurs repartent à zéro pour la nouvelle période.

Mais l'historique reste accessible.

====================================================================
9. EVOLUTION DE LA BOUTIQUE
====================================================================

La boutique doit devenir un véritable système de distribution.

Workflow étudiant :

1. Ouvre la boutique
2. Choisit un article
3. Passe commande
4. Reçoit un reçu PDF

Workflow gestionnaire :

1. Reçoit une notification
2. Voit la commande
3. Prépare l'article
4. Peut relancer la notification

Workflow retrait :

1. L'étudiant présente son reçu
2. La gestionnaire vérifie
3. La commande passe à LIVREE
4. Le stock est mis à jour

Le système doit :

- générer un reçu PDF ;
- suivre les stocks ;
- suivre les retraits ;
- suivre les ventes ;
- suivre les mouvements financiers.

====================================================================
10. CONTRAINTES IMPORTANTES
====================================================================

Ne pas casser :

- le système actuel ;
- les modèles existants ;
- les paiements ;
- la caisse ;
- les salaires existants ;
- les rapports existants.

Préférer :

- extensions ;
- nouveaux services métier ;
- migrations sûres ;
- compatibilité ascendante.

Objectif final :

Transformer le dashboard gestionnaire en un système intelligent qui automatise les préparations, sécurise les opérations et laisse à la gestionnaire un rôle de contrôle, validation et pilotage.
