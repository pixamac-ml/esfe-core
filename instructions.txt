Analyse entièrement la configuration realtime/WebSocket du projet Django avant toute modification.

Objectif :
Corriger proprement les erreurs suivantes sans casser l’existant :

* 404 sur `/ws/communication/notifications/`
* erreurs HTTPS sur le serveur de développement Django
* websocket notifications non connecté
* fallback notifications actif mais websocket indisponible

Contexte :
Le projet utilise Django + Channels + système de communication interne centralisé.
Les workflows principaux (admissions, emails, paiements, notifications classiques) fonctionnent déjà correctement.
Le problème concerne uniquement la couche realtime websocket.

IMPORTANT :

* Ne pas casser les workflows admissions/paiements/emails déjà fonctionnels
* Ne pas modifier brutalement le système de communication
* Analyser avant toute modification
* Respecter l’architecture existante
* Préserver la compatibilité HTMX/AJAX existante
* Ne pas introduire de complexité inutile
* Priorité à la stabilité

PHASE 1 — AUDIT COMPLET

Analyser entièrement :

* asgi.py
* settings.py
* communication/
* communication/realtime/
* communication/consumers/
* communication/routing.py
* templates utilisant websocket notifications
* JavaScript websocket frontend
* Channels configuration
* Redis configuration éventuelle
* middleware websocket
* AuthMiddlewareStack
* URLRouter
* websocket_urlpatterns

Identifier précisément :

* pourquoi `/ws/communication/notifications/` retourne 404
* si le consumer existe réellement
* si les routes websocket sont bien enregistrées
* si ASGI est réellement actif
* si runserver utilise bien ASGI
* si le frontend utilise `ws://` ou `wss://`
* si le navigateur force HTTPS
* si la logique fallback notifications est correcte

PHASE 2 — CORRECTION PROPRE

Corriger uniquement ce qui est nécessaire.

Objectifs techniques :

* websocket notifications fonctionnel
* aucune erreur 404 websocket
* compatibilité développement local HTTP
* compatibilité future HTTPS production
* gestion automatique ws/wss selon environnement
* système stable et minimaliste

Le frontend doit utiliser une logique dynamique :

* `ws://` en développement HTTP
* `wss://` en production HTTPS

Vérifier aussi :

* channels est installé correctement
* ASGI_APPLICATION correct
* routing websocket branché dans asgi.py
* consumer notifications opérationnel
* imports corrects
* aucune boucle ou reconnect abusive

PHASE 3 — VALIDATION

Après correction :

IMPORTANT :
